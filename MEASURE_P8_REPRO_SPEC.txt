Measure P8 Reproducibility Specification

Purpose
Make `measure` usable and understandable by a third party without relying on
the original author's memory, local habits, or private workflow knowledge.

Why P8 Matters
- The software has improved internally through `P2`, `P3`, `P6`, and `P7`.
- A paper-quality software artifact also needs a reproducible entry point.
- Another researcher should be able to understand what to run, what inputs are
  required, and what outputs to expect.

Scope
P8 focuses on the smallest reproducible and explainable path for `measure`.
It is not meant to document the full research program at once.

P8.1 Define an Official Minimal Example
- Select one example case to serve as the reference usage path for `measure`.
- Keep it as small and clear as possible.
- Prefer a case that already behaves well with the current refactored code.

P8.2 Document Required Inputs
- Explain what files `measure` expects.
- Clarify the role of:
  - `config.yml`
  - `parameters.txt`
  - case directories
  - extracted meltpool files where relevant
- State physical units for the key parameters.

P8.3 Document Key Parameters
- Explain in plain language what the important `measure` settings do.
- Include:
  - `y_begin`, `y_end`
  - `x_min`, `x_max`
  - `cell_size`
  - `min_points_per_zrow`
  - trimming options
- Explicitly state that `spot size` means laser diameter.

P8.4 Document Expected Outputs
- Explain the main output files generated by `measure`.
- Include:
  - `row_statistics.csv`
  - `cross_sections_statistics.csv`
  - continuity information
  - generated plots/images where relevant
- Explain what each output is for.

P8.5 Document Expected Runtime Messages
- Explain the meaning of:
  - clean configuration errors
  - nominal-window adjustment warnings
  - trimming snap warnings
- Clarify which warnings indicate adaptation rather than failure.

P8.6 Provide a Short Reproduction Workflow
- Document the shortest realistic sequence for running `measure`.
- Keep it command-focused and practical.
- Example structure:
  1. install package
  2. inspect example inputs
  3. run `simtopc measure config.yml`
  4. inspect outputs

P8.7 Keep the First Reproducibility Pass Small
- Do not try to package every simulation or every dataset immediately.
- Start with one trustworthy example and one clear workflow.

Desired Outcome
After the first pass of P8:
- another person can understand how to run `measure`
- another person can understand what inputs and outputs mean
- the repo is less dependent on author-specific knowledge
- the software looks more like a publishable artifact

Suggested First Implementation Steps
1. choose the official minimal example for `measure`
2. identify which files from that example are strictly required
3. write a short reproducibility guide
4. cross-check that the documented commands and outputs match reality
